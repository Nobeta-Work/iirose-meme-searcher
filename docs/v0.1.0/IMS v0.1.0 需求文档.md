# IMS v0.1.0 需求文档

## 1. 文档信息
- 项目名称：iirose-meme-searcher（IMS，蔷薇表情包搜索器）
- 文档版本：v0.1.0
- 文档日期：2026-03-21
- 当前状态：需求冻结（可进入开发）
- 目标：在 iirose 中提供“前缀触发搜索、点击即发”的表情包体验，并支持插件导入与前端配置。

## 2. 原始需求对齐
基于 `需求文档.md`，v0.1.0 需求范围明确为：
- 模仿 QQ 表情包交互：用户输入字符后可调起表情包搜索器。
- 展示可选表情包图片列表，点击后直接发送。
- 以插件方式接入，支持通过 JS 链接导入（示例：`https://shadowsee.icu/upload/iirose-scdn.js`）。
- 提供 iirose 适配前端面板。
- 支持可配置项（至少包含前缀关键词）。

## 3. 本版本确认结论（你我已确认）
- 表情源：来自 Bing 搜索结果中的远程图片 URL。
- 检索模式：基于 Bing 搜索结果进行实时检索，不要求用户自备付费能力或 API Key。
- 触发方式：固定前缀触发（示例：`/m`），同时支持 `/m关键词` 与 `/m 关键词`。
- 发送方式：点击候选后以裸图片 URL 发送。
- 发送范围：仅公屏消息。
- 界面形态：配置面板与搜索候选区分离；配置面板仅作为设置入口，搜索候选区为“输入框锚定的非模态悬浮候选条”，通常呈现在输入框上方。
- 兼容范围：v0.1.0 仅承诺 iirose 网页端。
- 空关键词策略：展示空状态，不展示最近使用或热门推荐。
- 交互方式：v0.1.0 仅承诺鼠标交互，不承诺键盘上下选择与回车发送。
- 配置入口：提供轻量配置入口，至少支持前缀关键词与搜索引擎配置；v0.1.0 默认内置 Bing。
- 检索前缀词：支持在配置面板中配置多个检索前缀词，搜索时按“前缀词 + 用户关键词”拼接查询。
- 候选数量：默认展示 8 个结果，可在配置面板中调整，范围限定为 1-20。
- 候选条关闭条件：前缀被删除、房间切换、发送成功时关闭或重置；输入框失焦不强制关闭。
- 发送后行为：发送成功后清空输入框。
- 性能要求：v0.1.0 不设硬性 SLA，以“可用、稳定、可感知反馈”为验收基线。

## 4. 范围定义（In Scope）
- 插件可通过 JS 链接导入并生效。
- 插件在 iirose 网页端可用。
- 在输入框输入前缀后触发表情搜索。
- 基于 Bing 搜索结果进行图片检索。
- 以输入框锚定的非模态悬浮候选条展示候选表情列表（图像预览），通常位于输入框上方。
- 点击候选后将内容发送至公屏。
- 提供基础配置能力：前缀关键词、搜索引擎可改。
- 提供检索前缀词配置能力，允许多个前缀词参与查询拼接。
- 在 iirose 前端界面中可正常使用，不破坏原有输入流程。

## 5. 非范围（Out of Scope）
- 私聊发送。
- 广播（弹幕）发送。
- 本地离线素材包导入与管理。
- 素材审核系统、版权审核系统。
- 多源聚合排序、高级推荐策略。
- 桌面封装版与移动端兼容承诺。
- 键盘上下选择、回车发送等键盘交互增强。

## 6. 用户流程（需求视角）
1. 用户在聊天输入框输入前缀（如 `/m`）。
2. 插件识别前缀并启动 Bing 图片检索流程。
3. 用户继续输入关键词，输入框附近出现匹配的横向候选图片列表；该列表为输入框锚定的非模态悬浮候选条，通常位于输入框上方。若关键词为空，则展示空状态提示。
4. 用户点击某一候选项。
5. 插件将该项裸图片 URL 发送到公屏。
6. 发送成功后，输入框被清空。
7. 用户获得发送结果反馈（成功或失败）。

## 7. 功能需求

### 7.1 触发与检索
- 仅在命中配置前缀时启动检索流程。
- v0.1.0 默认且唯一内置的搜索引擎为 Bing。
- 检索结果来源于远程搜索引擎返回的图片结果。
- 前缀后可跟随关键词进行筛选，同时支持 `/m关键词` 与 `/m 关键词` 两种输入形式。
- 查询词应由“检索前缀词列表 + 当前关键词”按空格拼接形成。
- 示例：当前缀词配置为 `duitang.com`、`表情包`、`白圣女` 时，输入 `/m 开心` 应检索 `duitang.com 表情包 白圣女 开心`。
- 搜索引擎为配置项，但 v0.1.0 当前唯一可选值为 Bing；保留该配置项是为后续多源扩展预留。
- 关键词为空时显示空状态提示，不展示最近使用或热门推荐。

### 7.2 候选展示
- 候选列表至少包含：缩略图、名称（或关键词摘要）。
- 候选列表为输入框锚定的非模态悬浮候选条。
- 候选列表通常位于输入框上方，并随输入框位置保持贴边关联。
- 候选列表默认展示 8 个结果。
- 候选列表支持横向滑动浏览。
- 前缀被删除、房间切换、发送成功时，候选列表应关闭或重置。
- 候选为空时必须展示空状态文案。
- 远程数据加载中时必须展示加载状态。
- 远程数据失败时必须展示失败状态与重试入口。

### 7.3 发送行为
- 点击候选即发送，不弹二次确认。
- 发送对象为公屏。
- 发送内容采用裸图片 URL。
- 发送成功后清空输入框。
- 发送后提供可感知反馈：成功轻提示，失败可重试。

### 7.4 配置能力
- 必选配置：触发前缀。
- 必选配置：搜索引擎。
- 必选配置：候选上限。
- 可选配置：检索前缀词列表。
- 应提供轻量配置入口，供用户在网页端完成上述设置。
- 配置面板仅作为设置入口，不承载搜索候选展示。
- 搜索引擎配置项在 v0.1.0 中当前唯一可选值为 Bing。
- 检索前缀词列表支持多个词条，建议以逐行输入方式维护。
- 候选上限默认值为 8，允许范围为 1-20。
- 配置项应具备本地持久化能力；刷新页面后的保留策略需与实现保持一致。
- 配置变更后应即时生效（无需重启页面为佳；如需刷新，应在文档明确）。

## 8. 界面与交互约束
- 配置入口为独立轻量面板，不与搜索候选列表复用。
- 搜索候选列表为非模态悬浮候选条，不得遮挡或抢占输入焦点。
- 搜索候选列表通常展示在输入框上方，并与输入框保持锚定关系。
- 候选列表不得遮挡用户输入焦点与核心发送控件。
- 候选列表应在常见窗口尺寸下可见且可操作。
- v0.1.0 仅承诺鼠标点击交互。
- 键鼠操作需保持直观：输入-浏览-点击发送路径最短。

## 9. 协议、可行性与导入约束（用于需求边界，不代表立即实现）
- 与 iirose 通信基于 WebSocket 机制。
- 社区文档显示消息与媒体可通过特定文本格式承载，v0.1.0 采用“裸图片 URL 发送”策略。
- 上游搜索引擎的原始响应结构不在 v0.1.0 需求中冻结；插件内部只需归一化为需求文档中的表情项模型。
- 协议与报文细节存在社区维护差异，落地前需通过实际抓包和回归验证确认。
- 调试阶段可使用 WebSocket 日志方法进行消息核对。

### 9.1 网页端前端可行性结论
- v0.1.0 在 iirose 网页端具备前端实现可行性。
- 可行性依据之一：`https://iirose.com/` 当前为网页应用，顶层页面存在 `iframe#mainFrame` 承载主界面。
- 可行性依据之二：社区已有浏览器侧 JS 脚本通过 Hook WebSocket、暴露 `send` 方法等方式对网页端行为进行观测和增强。
- 可行性依据之三：社区存在第三方 CSS 注入与分发实践，说明网页端存在前端增强与样式扩展空间。
- 以上结论基于社区实践与页面结构推断，不代表存在官方稳定插件 API。
- 需求侧收口：v0.1.0 允许采用 DOM 注入、输入监听、样式覆盖等网页前端手段实现候选条与配置入口。

### 9.2 远程 JS 导入方案（收口）
- v0.1.0 交付形态收口为：单一远程 HTTPS JS 文件。
- 该文件应为可直接执行的经典脚本（classic script / IIFE），不要求浏览器支持 ESM `import()`。
- 用户导入方式收口为：在 iirose 的 JS 终端执行固定安装器片段，由安装器定位当前有效页面上下文（当前窗口或承载主界面的 `mainFrame`），再注入远程 `<script src=\"...\"></script>`。
- 远程脚本需具备重复加载保护，至少暴露一个全局版本标记或实例标记，避免重复初始化。
- 远程脚本地址应使用 HTTPS，服务端应返回可执行 JS 内容类型（如 `text/javascript`）。
- 安装说明中应提供唯一推荐的一行式导入语句；需求文档冻结导入机制，不冻结最终发布域名。
- 页面刷新后脚本实例视为卸载；再次使用时重新执行导入语句。

## 10. 数据模型（需求级最小集合）

### 10.1 表情项
- `id`：唯一标识。
- `name`：展示名称。
- `keywords`：检索关键词数组。
- `url`：图片地址。
- `enabled`：是否可用。

### 10.2 插件配置
- `triggerPrefix`：触发前缀。
- `searchEngine`：搜索引擎标识。
- `keywordPrefixes`：检索前缀词列表，按顺序参与查询拼接。
- `maxCandidates`：候选上限，默认 `8`，取值范围 `1-20`。

## 11. 异常与边界场景
- 前缀命中但关键词为空。
- 远程数据超时或不可达。
- 返回结果中存在无效 URL。
- 搜索配置变更后，候选结果与当前输入状态的一致性。
- Bing 搜索结果页结构变化、频控或反爬导致结果不可用。
- 用户连续快速点击同一候选导致重复发送风险。
- 房间切换后候选列表状态与输入状态一致性。

## 12. 验收标准（v0.1.0）
- 可通过 JS 链接导入插件，并在 iirose 网页端进入可用状态。
- 输入前缀后可稳定触发搜索流程。
- 同时支持 `/m关键词` 与 `/m 关键词` 两种触发输入形式。
- 已配置检索前缀词时，查询词拼接结果正确。
- 输入关键词后，可展示与输入框锚定的横向候选列表，视觉效果接近 QQ 表情候选体验。
- 候选数量默认展示 8 个，且可在配置面板中调整为 1-20。
- 空关键词时可展示空状态。
- 点击候选后可向公屏发送对应裸图片 URL。
- 发送成功后输入框被清空。
- 删除前缀、房间切换、发送成功后三种情况下，候选列表状态正确关闭或重置。
- 失败、空结果、加载中三类状态均可被用户识别。
- 至少支持前缀关键词、搜索引擎、候选上限配置并可生效；v0.1.0 当前唯一可用搜索引擎为 Bing。

## 13. 风险与假设
- 风险：不同客户端环境下 UI 注入点和样式兼容存在差异。
- 风险：搜索引擎的返回结构、可访问性和更新策略可能变化。
- 风险：Bing 搜索结果结构、访问策略或可用性变化，可能直接影响插件搜索能力。
- 风险：远程资源稳定性、可访问性、图片失效不可完全控制。
- 风险：社区文档与线上行为可能存在偏差。
- 假设：v0.1.0 仅面向“可用 MVP”，优先保证流程闭环。

## 14. 版本后续（非 v0.1.0 承诺）
- 私聊发送支持。
- 多前缀与多源搜索。
- 桌面封装版与移动端适配。
- 键盘导航、回车发送等增强交互。
- 收藏、历史记录、最近使用排序。
- 更细粒度性能指标与监控埋点。
